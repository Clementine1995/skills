# AGENTS.md

## 职责

本文件是 AI 进入仓库后的第一入口，包含强约束、任务分级、工程风格、架构红线填写位和证据收口。

## 必读

进入仓库后，必须先读取：

1. `README.md`：项目总说明、文件地图、最小启动和环境口径。
2. 本文件：AI 协作约束。

## AI 协作材料

- 本仓库不保存临时 AI 协作过程材料或历史过程归档。
- 复杂协作过程材料必须由用户明确指定仓库外目录或外部工程承接；本仓库只保留业务源码、项目说明、环境/发布说明和必要验证脚本。

## 任务分级

### 只读核查

只允许读取文件、搜索文本、查看状态、查看差异、读取日志、读取配置、执行数据库 `SELECT` 和调用不会改变状态的查询接口。

只读核查不得写文件、写数据库、执行迁移、安装依赖、重启服务、提交 Git 或推送 Git。

数据库只读查询前必须说明环境、库名和影响范围；只读澄清优先只执行普通 `SELECT`。如数据库支持只读事务，可用 `START TRANSACTION READ ONLY`（或等价语法）后执行 `SELECT`，最后用 `ROLLBACK` 结束只读事务。这里的 `ROLLBACK` 只表示结束只读查询事务，不是业务数据回滚；事务内禁止任何写入、迁移、DDL、DML 或状态变更语句。

### 小改动

允许修改当前任务直接相关的代码、测试或文档，并运行本地匹配范围验证。

小改动不包含数据库写入、迁移、远端服务状态变化、依赖安装、真实业务写接口调用或生产发布。

### 真实联调

涉及真实接口调用、数据库写入、worker、浏览器自动化、服务状态检查或跨系统端到端验证时，属于真实联调。

真实联调执行前必须说明环境、对象、影响范围和验证证据。

本地端到端默认只在主开发分支和开发环境配置下执行。测试环境发布使用测试分支和测试配置，生产发布使用生产分支和生产配置。生产发布后默认只做发布和只读可用性检查，不默认运行远端 worker 或写生产切面数据。

### 生产发布

测试环境发布和生产发布必须按 `README.md` 的环境口径执行。

执行计划和对话口径必须区分测试环境发布和生产发布。不得把测试环境发布当作生产发布。

不得用本地仓库之间的临时合并、远端直接改代码、测试分支合并到生产分支或绕过标准合并方向的方式替代标准流程。

合并前必须先从远端拉取最新源分支和目标分支，让本地目标分支与远端目标分支对齐后再合并远端源分支。合并冲突只能按目标环境口径解决，保留目标环境特有的数据库、接口地址和端口。禁止将测试分支合并到生产分支。

AI 不允许使用 Git 回滚脚本或回滚命令修复流程错误（包括 `git reset`、`git reset --hard`、`git checkout <commit> -- <path>`、`git restore`、`git revert` 等）。发生错误时必须停止，输出当前分支、HEAD、工作区状态、已执行命令和可选人工恢复方案。

数据库写入、软删除、数据修复、迁移和回滚都必须单独获得明确授权。执行前必须说明目标表、筛选条件、影响范围、回查 SQL 和失败后的处理方式。

## 必须

- 写文件前，先说明要改哪些文件和目标。
- 改动范围必须收敛，只处理当前任务。
- 修改前必须界定本次目标行为和废弃行为，只沿实际调用链修改当前任务直接相关的入口、服务、client、模型、测试和文档。
- 凡是准备执行删除、清空、覆盖、替换、移动、撤回、回滚等可能破坏现状的操作前，必须先只读查询确认目标、范围和影响；涉及历史数据、日志、配置、远端目录等状态性内容时，必须停止并说明，不能自由发挥直接执行。
- 如果旧逻辑已被废弃，必须删除旧实现、旧分支、旧测试期兜底、旧文档描述和相关测试，不保留新旧双轨；只有用户明确要求兼容期时，才允许保留旧逻辑，并必须说明兼容范围、触发条件和后续删除点。
- 说明项目事实、接口、问题、修改建议时，尽量给出文件和行号。
- 改动后必须说明改了什么、验证了什么、还剩什么风险。
- 涉及数据库、迁移、远端服务、真实接口调用时，先说明环境和影响范围。
- 结论必须区分事实、推断和未验证风险。

## 禁止

- 未经用户明确授权，不执行 `git add`、`git commit`、`git push`、`git pull`。
- 未经用户明确授权，不写数据库、不执行迁移。
- 未经用户明确授权，不启动、停止、重启远端服务。
- 未经用户明确授权，不删除文件或目录。
- 未经用户明确授权，不安装、升级、移除依赖。
- 未经用户明确授权，不修改环境变量或运行配置。
- 不为了降低改动量而留下已废弃的兼容代码、fallback、开关、临时映射、隐藏入口或永远不会再走的分支。
- 不做与当前任务无关的顺手重构，不扩大接口能力，不改无关命名、格式、依赖、配置、迁移或部署口径。
- 不把本地验证授权、测试库授权或只读核查授权扩大到生产发布、远端 worker 或生产切面写入。
- 不在远端直接修改业务代码后绕过本地仓库。

## 默认允许

- 读取文件。
- 搜索文件、目录、文本。
- 运行只读命令，例如文本搜索、目录列举、`git status`、`git diff`、`git log`。
- 查看数据文件（CSV、JSON、日志、Markdown、源码文件片段）。
- 解释实现、比较方案、指出风险、提出文档整理建议。

## 工程风格

> 占位符 `{{package_name}}` 为主代码包名，按实际替换。

### 语言与文本（违反 = 返工）

1. 代码标识符（模块、类、函数、变量、表名、字段名）必须使用英文。
2. 代码注释、docstring、面向业务的说明文字必须使用中文。
3. 中文文档必须使用 UTF-8 编码保存。
4. Git commit message 必须使用中文，前缀必须是以下之一，禁止其它前缀：`feat:`、`fix:`、`docs:`、`refactor:`、`test:`、`chore:`。
   > 团队可按需增删前缀，确定后禁止使用列表外前缀。

### 业务注释（违反 = 返工）

5. 业务注释统一使用中文 `#` 注释（或当前语言的单行注释语法）。
6. 文件职责注释放在文件顶部可见区域，说明当前文件承担的业务边界。
7. 方法职责或逻辑概览注释放在函数或方法定义上方。
8. 函数内部大步骤注释放在对应代码块上方，说明业务阶段、状态流转或关键处理顺序。
9. 关键取值、关键分支、关键校验、异常语义、硬规则和软约束来源，必须在就近位置用注释说明业务原因。
10. 同行注释只用于必须贴近单行表达式说明的关键取值或关键分支；默认优先写在代码上方。
11. docstring 不承载业务规则、状态流转、硬规则、软约束或长段逻辑说明。
12. 注释解释业务逻辑、字段含义、状态流转、异常语义和规则来源，不解释显而易见的语法。
13. 注释只写当前有效事实，不写历史过程、不写新旧对照、不描述已废弃逻辑。
14. 在既有逻辑中的点状修改，必须在修改点附近说明该改动改变的业务行为、数据流或状态边界。
15. 复杂链路注释必须服务人工顺调用链查证，优先放在正式入口、批量触发点、并发边界、外部接口调用点、状态回写点、结果拆分或合并点、业务主键映射点。
16. 注释不得作为兼容说明保留已废弃入口、旧分支、兜底路径或双轨行为；废弃行为必须在代码和测试中同步清理。

### 代码风格（违反 = 返工）

17. 导入语句必须使用从 `{{package_name}}` 包开始的绝对路径。禁止任何形式的相对导入。
    > 若使用不支持包名前缀的语言，改为"导入路径必须从模块根开始"的等价约束。
18. 禁止创建以下文件名的模块：`utils`、`common`、`helpers`、`misc`、`tools`。
    > 按实际技术栈调整扩展名，例如 `utils.py`、`utils.ts`。
19. 新增源码文件名必须包含动词或具体名词。禁止仅以形容词或泛指词命名。

### 测试命名（违反 = 返工）

20. 测试函数名禁止包含：数字日期、任务编号、审查日期。
21. 测试函数名必须描述被测行为或被测场景，不描述实现细节。

## 架构红线

> 本节为项目特有红线，换项目要重写。每条红线应可被测试或 grep 验证，宁缺毋滥。
> 下方按维度给出填写方向，按项目实际保留或改写，删除不适用的章节。

### 项目定位

- 填写本项目一句话定位、负责什么、不负责什么。

### 分层与依赖方向

> 示例方向：
> - 路由层禁止出现数据库访问代码。
> - 领域层禁止反向依赖路由层。
> - 外部 HTTP/RPC 调用必须经过独立的 client 层，业务层禁止直接使用 HTTP 库。

### 入口与操作限制

> 示例方向：
> - 禁止新增特定 HTTP 方法 / 路径入口。
> - 正式业务入口仅限列举范围，禁止新增其他正式入口。

### 文件命名约束

> 示例方向：
> - 某目录下文件名必须匹配特定模式。
> - 禁止泛指命名文件。

### 数据完整性

> 示例方向：
> - 软删除查询必须包含 `is_deleted = 0`（或等价条件）。
> - 失败记录的关键字段禁止为 NULL。

### 可观测性

> 示例方向：
> - 正式入口必须生成链路追踪标识（如 request_id）并绑定日志上下文。
> - 同一链路内各层日志必须携带同一链路追踪标识。

### 测试约束

> 示例方向：
> - 涉及表结构或迁移改动后，必须运行 head 检查（或等价命令），输出必须唯一。
> - 架构边界测试必须放在指定测试目录。

## 证据口径与收口

### 小改动证据

- 本次修改文件。
- 本次关键 diff。
- 匹配范围验证结果。
- `git status` 状态。

### 接口链路证据

- 入口接口或 worker。
- 请求关键字段。
- 响应关键字段。
- service、client 和下游接口文件位置。
- 数据库只读回查结果。
- 日志事件、时间和链路追踪标识。

### 端到端证据

本地端到端证据必须来自主开发分支和开发环境配置。测试环境或生产发布证据不得替代本地端到端证据。

端到端证据固定为四方：前端浏览器自动化结果、接口 smoke、数据库只读回查、审计日志。关联链路追踪标识、批次标识、任务标识或业务主键。

### 部署证据

- 本地分支和 commit。
- 测试分支和 commit（如有测试环境）。
- 远端服务目录。
- 容器或进程状态。
- 健康检查。
- 关键日志。

### 临时产物

- 一次性排查脚本、临时 SQL、浏览器自动化截图和临时测试记录默认不提交。
- 复杂协作过程材料默认不提交；如需保存，必须由用户明确指定仓库外目录或外部工程。
- 可复用验证脚本可保留在项目的临时验证目录，但必须有明确用途和稳定入口。
- 临时产物是否提交，以用户明确要求为准。

### 收口输出

完成任务后必须说明：

- 改了什么。
- 验证了什么。
- 没有验证什么。
- 剩余风险是什么。
- 是否已提交和推送。

## 字符集

- 中文文件按 UTF-8 读取和写入。
- 读取中文文件时注意指定正确的编码，避免乱码。
